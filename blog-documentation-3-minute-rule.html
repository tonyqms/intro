I spent two days writing a 15-page pipeline manual. Two people read it. Nobody used it.

The 3-Minute Rule

If someone can‚Äôt find what they need in three minutes, the documentation has failed. Most people arrive at docs in a mild state of panic: a run is stuck, a deliverable is due, or a cluster is burning money. They don‚Äôt want a textbook. Branch 1: Execution Docs (The ‚ÄúHow‚Äù)

Execution docs live next to the code and optimize for time-to-first-success, not technical completeness. They reduce support tickets by eliminating ambiguity at the most common failure points.

1) Quick Start: 30 Seconds to First Run

The Quick Start is intentionally minimal: one sentence on what the pipeline does, one command to run it, and one pointer to outputs. The psychological win of ‚ÄúI got it to run‚Äù is enormous.

Example:

2) Common Tasks Cheat Sheet

Most users repeat a small set of operations. Capture the 20% that generates 80% of support traffic.

  • Override default parameters
  • Resume a failed run
  • Check status on the cluster or cloud
  • Find logs and outputs quickly
  • Know who to contact when it still fails

3) ‚ÄúIt Broke‚Äù Troubleshooting

Troubleshooting should be ordered by frequency and cost, not alphabetically. Each entry should include a symptom, a copy-paste fix, and a second step if needed.

  1. Out-of-memory / OOMKilled
  2. Input file not found or wrong pattern
  3. Permission denied / auth issues
  4. Disk space / quota problems
  5. Misconfigured parameters

Execution Doc Layout

In practice, I keep execution docs short and close to the code:

  • README.md: what it does, quick start, outputs, links
  • CHEATSHEET.md: common tasks with real commands
  • TROUBLESHOOTING.md: top failures with fixes

I use a short template to keep pages tight:

In regulated environments, I keep SOPs thin: roles, responsibilities, checkpoints, and evidence. Use checklists and flow diagrams instead of prose, and keep SOPs under version control.
